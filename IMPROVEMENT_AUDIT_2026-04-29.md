# 改善点洗い出し（2026-04-29）

## 優先度A（今すぐ着手推奨）

1. **ESLintを非対話で実行できる状態にする**
   - 現状 `npm run lint` で Next.js の ESLint 初期設定プロンプトが表示され、CI/自動検証が止まる。
   - `.eslintrc`（または `eslint.config.*`）をリポジトリに明示追加し、`next lint --no-cache` が即時実行できるようにする。
   - これにより品質ゲート（PR時の自動チェック）を構築可能になる。

2. **READMEの進捗と実装状況ドキュメントの不整合解消**
   - `README.md` は MVP 未完了チェックが多数残る一方、`CURRENT_STATUS.md` では認証やデータセット機能が完了扱い。
   - 同一情報源（例: `CURRENT_STATUS.md`）を正として、READMEのフェーズ表を同期する。
   - ドキュメント不整合は開発優先順位の誤解を生むため、最優先で更新する。

3. **テスト基盤の導入（最低限の回帰防止）**
   - `package.json` にテストスクリプト（`test`）が未定義で、変更時の安全性確認が困難。
   - まずは API ルートとバリデーション層に対し、Vitest/Jest + Testing Library などで最小テストを追加。
   - 目標: 認証、データセット取得、アップロード周辺の主要フローを自動テスト化。

## 優先度B（短期改善）

4. **環境変数の必須/任意を実行時に厳格化**
   - `.env.example` への言及はあるが、実行時の必須項目のバリデーションルールが読み取りづらい。
   - `lib/env.ts` を中心に「開発/本番で何が必須か」を明文化し、起動時に早期失敗させる。

5. **未実装機能のUI表現を統一する**
   - 「UIのみ実装」「将来実装」などの状態が複数文書に分散。
   - 画面上で `Coming Soon` バッジ、または機能フラグで明示し、ユーザー期待値を管理する。

6. **決済/購入フロー着手前に設計ドキュメントを1枚化**
   - 現状、購入系は優先度が高いが未実装。
   - 先に「注文状態遷移」「返金/キャンセル」「手動決済とStripeの分岐」を1ドキュメントに固定すると、実装の手戻りが減る。

## 優先度C（中期改善）

7. **運用監視と障害調査の標準化**
   - 監視基盤（Sentry等）の記述はあるが、実装状態が不明瞭。
   - APIエラーの共通フォーマット化、トレースID付与、通知ルール（閾値）を定義する。

8. **パフォーマンス最適化の計測先行**
   - まず Lighthouse / Web Vitals のベースライン計測を行い、改善対象を定量化。
   - 画像最適化、一覧ページの取得戦略（キャッシュ/ISR）を段階的に改善。

9. **データ拡張機能はPoC範囲を限定**
   - AI拡張は魅力だが実装コストが高い。
   - まず画像のみ・単一モデル・単一ワークフローでPoC化し、価値検証後に横展開する。

  - READMEとCURRENT_STATUSの同期
  - 最小テスト（2〜5本）追加
- Week 2:
  - 購入フロー設計書（状態遷移）を確定
  - 決済実装のスパイク（手動決済→Stripeの順）
